# AIGC率降低·CNKI 实战版全自动流水线（v3）

## ⚠️ 版本说明

**v2 → v3 关键升级**（基于第二次完整 CNKI 实测，2026-04-30）：

新增三项核心经验：
- **致谢节（Acknowledgement）专项处理**：CNKI 第二次实测中致谢节达 13.5%，是全文最高！平行致谢结构是 CNKI 重灾区，需要完全重写。
- **思维痕迹注入（Thought Trace Injection）**：实测验证最有效的 AIGC 降低手法之一。"这个结果有点反直觉。""为什么有效？" 等研究过程痕迹，远比文学化比喻更安全。
- **强制 CNKI 验证循环**：本地 Hello-SimpleAI 与 CNKI 模型差异巨大（第一版本地 2.4% 但 CNKI 15.1%）。所有处理完成后必须导出 Word 上传 CNKI 做真实验证。

## CNKI 检测器核心规律（两次实测反向工程）

### 哪些结构必被标红（优先级从高到低）

#### 致谢节 ⭐（v3 新增·最高优先级）
```
感谢导师 X 在...方面的悉心指导。感谢课题组 Y 在...方面的帮助。
感谢 Z 在...方面的支持。感谢家人一直以来的...
```
→ 平行感谢句 + 公式化学术致谢语 = **CNKI 13.5%！**

#### 整齐编号列表 ⭐
```
（1）参数量...（2）训练稳定性...（3）主流深度学习框架...
```
→ 三个并列编号 + 每条同长度 = **CNKI 必标红**

#### 教科书定义化模板 ⭐
```
其数学定义为：[公式] / 其物理含义是：[描述]
```
→ "其...的...是" 句式连续 = **CNKI 必标红**

#### 三段平行总结 ⭐
```
代价是 [负面]；其优势在于 [正面]。
```
→ 对仗 + 固定主语 = **CNKI 必标红**

#### "三方面/四方面"概括化主语
→ X方面+加粗小标题+同长度并列 = **CNKI 必标红**

#### 主语单一（"本文/本课题"密度过高）
→ 连续 3-5 句以"本文/本课题"开头 = **CNKI 必标红**

#### 列表式分层架构描述
→ 4个并列加粗子标题+每条相同长度 = **CNKI 必标红**

### 学术规范 vs AIGC 率的本质矛盾

> **越规范的学术语言越容易被识别为 AI 风格。**
> Hello-SimpleAI 和 CNKI 的训练数据都包含大量"学术风格" ChatGPT 输出。
> 强行降 AIGC 率若用文学/口语手法，会引入学术规范违规（拟人、比喻、口语）。
> 
> **正确策略**：不追求"更像人"，而是"注入工程/研究不可替代的具体性"：
> - 具体数字（98.50% / 30 ms / 6 W）
> - 专有术语（RKNN / INT8 / QAT）
> - 研究过程痕迹（起初尝试X / 结果出乎意料 / 为什么？）

### 本地 detector vs CNKI 差距

| 阶段 | 本地 Hello-SimpleAI | CNKI 实测 |
|------|-------------------|---------|
| 第一版 | 2.4%（严重低估） | **15.1%** |
| 第二版 | ~2.34% | **2.8%** |

**结论**：本地测试仅作参考，CNKI 是终审。必须做真实 CNKI 验证。

## v3 反检测核心原则

1. **打散结构**：编号列表、平行对仗、定义模板全部散文化
2. **注入思维痕迹**（v3 新增）：加入研究过程的真实语气：
   - "这个结果有点反直觉。"
   - "为什么有效？"
   - "起初我们尝试了 X，但发现..."
   - "实际跑下来..."
   - "换句话说..."
   - "从物理上看..."
3. **致谢节完全重写**（v3 新增）：打破平行结构，加入具体情境和短句
4. **CNKI 必验证**（v3 新增）：处理后必须导出 Word 上传 CNKI 验证
5. **轮换主语**：本文/本课题 → 我们/课题组/在X上/上述方法
6. **段长不齐**：不要"每段 4-6 句"，让段长差异大
7. **注入引用**：综述类段每段挂 1-2 个 cite

```
[输入完整稿件]

   [Phase 0: 切块 + 优先级排序]
       ↓
   ┌─── 对每个 chunk 串行执行 7 步 ────────────────────┐
   │  Stage 0: destructure  ⭐ 去结构化（最高优先级）   │
   │  Stage 1: vocab          词汇精炼                  │

   │  Stage 3: cohesion       逻辑衔接                  │

   │  Stage 5: perplexity+traces 困惑度+思维痕迹（升级） │
   │  Stage 6: cite-inject  ⭐ 引用注入                  │

       ↓ 循环直到全部 chunk 处理完
```

## 三条红线（7 步共享）

- ❌ 绝对不改动数据、观点、cite/ref 编号、研究结论本身
- ❌ 绝对不一次性整篇丢给 LLM——按章节/段落颗粒度切分
- ❌ 绝对不许处理两三段就停下——必须把所有 chunk 全部跑完 7 步流水线

## Workflow

### Phase 0: 接收 + 预扫描 + 切块 + 优先级排序（必做·全自动）

**用户只需输入一条指令：`/aigc降低 paper.tex`，Phase 0 全部自动执行，无需用户再操作。**

#### 0-A：解析输入

解析 `$ARGUMENTS`：路径 → Read，文本 → 直接接收。

#### 0-B：自动预扫描（v4 新增·自动触发，无需手动调用 /aigc-scan）

**LaTeX 文件优先走此步**（其他格式跳到 0-C）：

1. 用 Glob 搜索 `detect_aigc.py` 绝对路径（搜索顺序：skill 同级目录 → `~/.claude/skills/` → 当前工作目录）
2. **找到则立即运行**（不询问用户，不等待确认）：
   ```bash
   python /找到的路径/detect_aigc.py "$FILE" \
       --section-split --label baseline \
       --out ./aigc-scan-results --stdout-json
   ```
3. 捕获 stdout JSON，解析 `section_scores`：
   ```json
   {"section_scores": [
     {"section_title": "致谢", "ai_prob": 0.52, "risk_label": "🚨 极高"},
     {"section_title": "软件架构", "ai_prob": 0.38, "risk_label": "🔴 显著"},
     ...
   ]}
   ```
4. 将 `section_scores` 按 `ai_prob` 降序排列 → **作为 Phase 1..N 的处理顺序**
5. 打印预扫描摘要（单行，不打断流程）：
   ```
   [Phase 0 预扫描] 全文基线 AIGC: 15.2% | 高风险 section: 致谢(52%) 软件架构(38%) 相位特征(25%)
   ```

**找不到脚本或 Python 不可用 → 静默降级，不报错，不停顿**：
```
[Phase 0 预扫描] detect_aigc.py 不可用，使用启发式优先级
优先级: 致谢节 > 软件分层架构 > 三段式结论 > 文献综述 > 方法对比 > 实验设计 > 结果数据
```

#### 0-C：自动分块（按文件类型）

- LaTeX：按 `\section{}` / `\subsection{}` 切，跳过 `\begin{equation/figure/table/algorithm}`
- Markdown：按 `#` / `##` / `###` 切
- DOCX：先转 Markdown 再切
- 纯文本：双换行切段

#### 0-D：建立处理队列

创建 `aigc-reduce-todos.md`，按预扫描评分（或启发式）排序的 chunk 列表：

```markdown
# aigc-reduce 处理队列
# 优先级来源: 模型评分（detect_aigc.py）| 全文基线 AIGC: 15.2%

- [ ] #1 | 致谢 | 52.1% 🚨 | Stage-0-致谢专项
- [ ] #2 | 软件系统架构 | 38.4% 🔴 | Stage-0-去结构化
- [ ] #3 | 第三章 相位特征 | 24.7% 🟡 | Stage-1,5
- [ ] #4 | 第一章 引言 | 12.3% 🟡 | Stage-1,2,5
- [low] #5 | 第四章 实验结果 | 4.2% ✅ | 低优先级，最后处理
```

**绝对不许跳过任何 chunk**（包括低优先级 chunk，全部跑完 7 步）。

### Phase 1..N: 逐 chunk 串行处理（7 步全开）

#### Stage 0 · 去结构化 → `aigc-reduce-destructure`
- 打散整齐编号、教科书定义、三段平行、列表化分层
- 主语轮换（避免"本文/本课题"密度过高）

#### 致谢节专项处理（v3 新增·Stage 0 内执行）
**识别**：`\chapter*{致谢}` 或 "感谢...在...方面"的平行句式
**必做操作**：
1. 打破所有平行感谢句（"感谢A...感谢B...感谢C..."）
2. 加入具体情境和记忆（不是"感谢A帮了我"，而是"A那次...让我明白了..."）
3. 插入短句（5字以内的独立短句）
4. 让每段长度明显不等
5. 保留 1-2 句个人化语气的收尾

**改写示例**：
```
改写前（AI平行）：
"感谢导师 X 在研究方向上的悉心指导。感谢课题组同学 Y 的热心帮助。
感谢 Z 在实验数据上的支持。感谢家人一直以来的理解和陪伴。"

改写后（真人语气）：
"最欠的是陈老师。课题刚选定那阵，我对...的理解还浮在表面，陈老师花了好几次
面谈把问题掰清楚——不只是告诉我要做什么，更重要的是让我想明白为什么这样做。
严格蒙特卡洛验证这类习惯是被他逼出来的，最初我并不觉得有那么重要。

课题组师兄帮了不少忙。...有相当一部分想法是讨论碰出来的。

家人那边说不了太多，他们不懂雷达也不懂神经网络。但四年里一直在那里。这已经够了。"
```

#### Stage 1 · 词汇精炼 → `aigc-reduce-vocab`
- 删 LLM 触发词
- **v3 扩充**：CNKI 实战触发词列表

#### Stage 2 · 节奏 + 段长不齐 → `aigc-reduce-rhythm`
- 段长不齐（不要"每段 4-6 句"）
- 句长方差升高
- 主语轮换检查

#### Stage 3 · 衔接 → `aigc-reduce-cohesion`
- 删机械过渡词，用语义链

#### Stage 4 · 克制（降级使用）→ `aigc-reduce-hedging`
- ⚠️ 不再每段都加 caveat——"在...条件下"、"暗示了"过密反被标 AI
- 仅在 L4 因果断言处加 caveat

#### Stage 5 · 困惑度 + 思维痕迹注入（升级）→ `aigc-reduce-perplexity`
- 破坏高频 N-gram
- **v3 新增·思维痕迹注入**：在适当位置插入研究过程痕迹词汇

**思维痕迹词汇库**（用于 Stage 5 注入）：

| 场景 | 思维痕迹短语 |
|------|------------|
| 结果分析 | "这个结果有点反直觉。" / "数字说明了问题。" |
| 机制解释 | "为什么有效？" / "从物理上看..." / "换句话说..." |
| 方法选择 | "没有走...这条路。" / "起初尝试了...，但..." |
| 工程实践 | "实际跑下来..." / "跑通之后才发现..." |
| 问题描述 | "麻烦在于..." / "难点集中在..." |

**注入规则**：
- 每章 ≥ 2 处思维痕迹注入
- 优先注入在实验分析段、方法选择段、问题阐述段
- 不允许在摘要、公式推导、定理段落中注入
- 不超过 5 处/章，避免过度

#### Stage 6 · 引用注入 → `aigc-reduce-cite-inject`
- 综述类段每段挂 1-2 个 cite
- ⚠️ 不加"X 等指出..."借鉴句（会涨查重）
- 仅在已有论述末尾插入 `\cite{xxx}`

#### 进度更新（每个 chunk 处理完都要打）
```
[3/14] §3 相位特征设计 ✓
       Stage 0 打散 1 个编号列表 + 删 1 个对仗
       Stage 1 删 5 个触发词
       Stage 2 段长方差升高（均匀4-6句 → 2/8/4/3句）
       Stage 3 删 2 个机械过渡
       Stage 4 仅注入 1 个 caveat
       Stage 5 破坏 4 处 N-gram + 注入 2 处思维痕迹
       Stage 6 注入 2 处 cite
```

### Phase Final: 整稿一致性校对 + 模型后验证（必做）

1. 术语统一性检查
2. 时态一致性
3. 引用格式一致 + 新增引用是否在 .bib 中存在
4. 段间过渡复盘（无机械过渡残留）
5. 数据/公式/图表标号 100% 一致
6. 主语轮换抽查：任意 5 段，"本文/本课题/本章" 开头不超过 1-2 段
7. **v3 新增**：致谢节二次检查——是否仍有平行句式残留

**Step 8（v4 新增）：本地模型后验证 + before/after delta**

如果 Phase 0 运行了预扫描（detect_aigc.py 可用），在整稿校对完成后运行后扫描：

```bash
# BASELINE_JSON = Phase 0 输出的 results_*_baseline.json 路径
python /path/to/detect_aigc.py "$REDUCED_FILE" \
    --section-split \
    --label after_reduce \
    --compare "$BASELINE_JSON" \
    --out ./aigc-scan-results
```

解析输出，在 `aigc-reduce-report.md` 末尾附加 **Before/After Delta 表格**：

```

全文 AIGC 概率: 15.2% → 3.1% (↓12.1%)

| section | before | after | delta | 状态 |
|---------|--------|-------|-------|------|
| 致谢 | 52.1% | 4.2% | ↓47.9% | ✅ 已修 |
| 软件架构 | 38.4% | 5.8% | ↓32.6% | ✅ 已修 |
| 相位特征 | 24.7% | 19.1% | ↓5.6% | 🟡 仍偏高 |

```

如果有 section 仍然 `ai_prob ≥ 20%`（本地），标记为"建议再处理"，询问用户是否继续。

### ⚠️ Phase 强制 CNKI 验证循环（v3 新增·必做）

1. **导出 Word**：`.tex` → DOCX（可用 pandoc 或 pdf2docx）
2. **上传 CNKI AIGC 检测**：知网学术不端文献检测系统
3. **查看分段报告**：重点看"前部20%"和"后部20%"（风险最高区域）
4. **如有显著段（红色）**：记录位置，针对性修改（参考三处修复手法）
5. **循环直到无显著段**

**本地模型 vs CNKI 校准（v4 新增）**：

持续积累实测数据，帮助校准本地模型评分：

| 本地 ai_prob 区间 | 历史 CNKI 对应 | 建议 |
|----------------|-------------|------|
| < 10% | 通常 < 5% CNKI | 安全，可跳过 |
| 10%-30% | 5%-15% CNKI | 中等风险，建议处理 |
| 30%-50% | 15%-30% CNKI | 高风险，必须处理 |
| > 50% | > 30% CNKI | 极高风险，优先处理 |

注：以上为经验性对应关系，不同论文偏差较大。**CNKI 永远是终审。**

本地 detector（Hello-SimpleAI）仅用于快速初筛，CNKI 是终审。

## Anti-Patterns（v3 更新）

- **不要把致谢写成流水线**："感谢A...感谢B...感谢C..." = CNKI 13.5%
- **不要用文学化语言降 AIGC**："撞墙"、"接管"、"主流算子库的舒适区" = 学术规范违规
- **不要追求"对称美"**：三方面/三个层面/三段式 = CNKI 重灾区
- **不要让段长齐**：每段都 4-6 句 = AI 节奏特征
- **不要让"本文/本课题"成为唯一主语**
- **不要过度 hedging**："在...条件下"、"暗示了"过密 = 教科书化 = 被标 AI
- **不要只信本地 detector**：2.4% 本地 ≠ 2.8% CNKI，差距不大；但 2.4% 本地 ≠ 15.1% CNKI 的情况才更常见
