# 输入整理指南

**适用场景**：用户给的不是填好的 `interview.md`，而是一段口语化文字、聊天记录、邮件
或者只是"这周做了 X、Y、Z"的零散描述。这种情况下，LLM 不能直接照抄，必须先按本指南
整理成可归类的信息，再喂给 Writer 阶段。

> 正常路径：`scan_project` 出 manifest → `interview_generator` 出问卷 → 用户填问卷
> → Writer 出周报。本指南是 fallback，仅当用户跳过问卷或 manifest 不足以支撑周报时使用。

## 常见原始输入

用户可能给你的不是标准工程结果，而是下面这些形式：

- 复杂的项目文件
- 大量的数据与图标
- 大量的功能错综复杂，以及大量的不再使用的僵尸代码文件，以及少量的核心代码
- 各种论文与报告
- 很口语化的一段描述

## 你的整理目标

先不要急着写周报，先把原始信息处理成以下几类：

1.不同的文件夹的功能是什么，彼此之间的联系与区别是什么

2.文件夹下的不同的代码的功能是什么，他是否重要，他是否是某个代码的历史版本，于上周相比他是否有变化

3.文件夹下的数据与图表是什么，她是否重要，它代表了什么含义，于上周相比有什么增删与变化

4.文件夹下的论文，报告是否有了推进，这个推进是否显著。是否有价值

对上述信息进行处理后，将信息进一步处理为以下四类：

1. 本周已完成
2. 本周推进中
3. 风险 / 问题 / 阻塞
4. 下周计划

## 整理规则

### 1. 同类事项先合并

如果多条记录在说同一件事，首先要清楚他们之间的联系与区别，如果有区别则要标注出来，没有区别则不要重复提及。

### 2. 先抓动作，再抓结果

优先识别：

- 代码做了什么改变
- 与上周相比推进了什么
- 交付了什么结果
- 解决了什么问题
- 卡在什么地方

### 3. 没完成的不算完成

如果某项工作还在中间状态，不要硬写成"已完成"，更适合写成：

- 已完成第一阶段
- 已推进到评审阶段
- 已输出初稿，待确认

### 4. 很碎的小事要归并

如果原始输入里有很多细小动作，优先按主题归并，不要逐条照抄。

例如，不要写：

- 改了标题
- 改了封面
- 调了排版

更适合写成：

- 完成文章发布前的内容和版式优化

### 5. 如果信息明显不够，先补问题

优先追问这些最少必要信息：

- 这周最重要的 2 到 3 件事是什么
- 哪件事已经完成，哪件事还在推进
- 有没有需要上级知道的风险或阻塞
- 下周最明确的计划是什么

### 6. 跟 manifest 对照

整理出每条事项后，对照 `manifest.json` / `diff.json`，看是否有 file diff 支撑：

- 用户说"做了 X" 但 manifest 里看不到对应改动 → 提示用户："系统未检测到对应文件改动，
  是否漏配 bucket roots？" 或要求用户补充具体改动了哪个文件。
- 用户没提到 X 但 manifest 显示 X 有显著 diff → 反向追问"X 这块是不是也想写进周报？"
